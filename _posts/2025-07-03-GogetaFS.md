# Abstract
以前去重系统的瓶颈是IO。存储技术的进步使得存储介质的访问延迟降低，吞吐量增加。现在哈希计算速度成为瓶颈，从加密哈希变成非加密哈希以实现更高效的指纹生成。

去重元数据的维护开销在高速IO路径中变得显著，占38%。

根本原因是FP2P映射：
1. 增加了持久化次数，持久化元数据
2. 为了保证一致性，需要立即刷盘

论文提出的解决办法：将FP2P映射和文件系统的L2P映射合并，形成LFP映射。这样就可以和文件系统元数据在一次IO中一起持久化。基于LFP映射对数据和元数据的管理来实现GogetaFS。

GogetaFS的特点：
1. 兼容
2. 高效
3. 内存节省

实验显示GogetaFS比现有的去重文件系统性能强，能够最小化元数据管理开销。

# Introduction
去重的经典实现是在文件系统之上建立一个单独的去重系统。

去重文件系统是将去重逻辑嵌入到文件系统。
优点：
1. 可以复用文件系统原有索引结构来简化数据重定向的实现。
2. 对应用程序来说是透明的，应用更广泛

去重文件系统的发展和局限如Abstract中所言，和理想设置还有很大的性能差距。

FP2P为保持一致性导致的开销：强制先写FP2P,再写L2P。要IO两次且限制了IO并行性。

LFP映射优点：
1. 重用文件系统的IO路径和崩溃一致性机制，简化了实现。
2. 避免了去重系统和文件系统的不一致性。

LFP映射带来了空间与性能的权衡：将FP2P扩展到L2P,会占用更多空间。
举例：
- 传统FP2P:100个相同逻辑块 → 1个FP2P条目 + 100个L2P条目指向它
- LFP扩展:100个相同逻辑块 → 100个LFP条目（每个包含重复指纹）

这种权衡不适用于传统去重文件系统：
1. 加密指纹尺寸大，会导致元数据体积暴增。
2. CPU计算开销才是主要瓶颈。
（只有在存储延迟是瓶颈时这种权衡才适用，否则使用LFP只会导致元数据体积增加，而带不来速度提升）

但是存储介质的升级和非加密哈希使得LFP映射成为一个适用的解决方案。

挑战：LFP会带来不兼容性
1. 数据管理：文件系统以extent的方式，去重以块级的方式---->用FP对应一个块范围
2. 元数据管理：？

# Background
- 独立的去重系统：设置L2P,FP2P两张表，还有在内存中的FP index
- 去重文件系统：合并了去重系统和文件系统的L2P表，对用户来说是透明的

# GogetaFS design
设计目标：
1. 通用性：GogetaFS的关键技术要能适应普遍的文件系统。
2. 高效性：块级去重和高效指纹搜索
3. 内容高效性：在各种应用场景中应具有内存效率。
4. 可调节性，关闭去重功能后不影响原有文件系统

组件：
1. LFP entry:由文件系统的L2P拓展而来
2. OFT:存储连续数据块的指纹
3. GLT:内存里的FP2P。使用RCU lock，适用于频繁的查找和比较。

GLT相比原来的FP2P表的优势：
1. 没有崩溃一致性要求
2. 部署在内存中，可以直接访问

OFT的优势：
1. 可以并行更新
2. 受益于空间局部性
3. 删除数据块时不用更新OPT
4. 利用了文件系统的崩溃一致性
   > GOGETAFS can add OFT updates to the journal together with LFP entry updates.
